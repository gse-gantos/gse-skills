# gse-drawing-engine (GSE fork)

> **Fork provenance:** forked 2026-07-10 from installed `drawing-engine` per ultraplan 4.2.
> Changes: canonical namespace (D1), map-driven input, `_Memory/` orientation replacing the retired
> MEMORY.md loader, supersession registration in `Claude/Map/`, backlink emission, map-update
> contract, dead `00_Resources/` references dropped. Pipeline, router, scripts, and protocols
> inherited unchanged.

**Supersedes `read-drawing-beta` and `drawing-library`** — one skill, one database,
one retrieval interface (canonical naming ruling: `_Memory/AGENTS.md` §9).

A drawing set is processed **once** into a structured, provenance-tracked,
coverage-honest file database (`drawing-db/`), and thereafter every output is
generated by **querying that database — never by re-reading the raw PDF.** Scripts
do the cheap deterministic work (split, text-extract, render, assemble, validate);
Claude does the understanding (classify, find references, flag coordination
issues, answer). JSON is the single source of truth; markdown is a regenerable
projection. Built and validated in phases (the source PRD is not present in this
workspace — the v1-vs-v2 lessons live in
`Claude/Platform/lessons_learned/drawing-skill-v1-vs-v2-comparison.md`); each
phase ships with an acceptance test under `scripts/test_*.py`.

## 0. Router — run this FIRST (PRD §4.1)

The mode is decided from **disk + coverage state, not the user's phrasing.** Before
doing anything else, resolve the target set and run the router:

```
python scripts/router.py --drawings "<path to PDF or drawings/ folder>" [--set-name "<set>"]
# or, when a set already exists:
python scripts/router.py --set-dir "<…/drawing-db/<set-name>>" --explain
```

It prints a JSON decision: `mode` (BUILD / UPDATE / QUERY), `reason`,
`load_references`, and a coverage/sheet summary. Three states:

1. **No catalogue** for the target set (`machine/manifest.json` absent) → **BUILD**
   (full process).
2. **Catalogue exists, a sheet is revised/new/stale** (manifest marks it `stale`;
   the hash-diff that sets that flag arrives in Phase 5) → **UPDATE** (reprocess
   only changed sheets).
3. **Catalogue exists and is current** → **QUERY**.

A catalogue whose run never finished (`run_status: building`, or sheets still
`pending`/`partial`) routes to **BUILD (resume)** — never a fresh rebuild.

### The deliberate seam (failsafe FS-1)
When QUERY hits a sheet that coverage marks `pending`/`failed`, it **does not
silently launch a heavy build.** It returns a `coverage_gap` signal naming the
missing sheets and lets the caller decide. Cheap reads of an already-processed
sheet are allowed; full (re)processing is never silent. This protects the
low-token guarantee. (Query-side enforcement lands in Phase 4.)

### Lazy reference loading (failsafe FS-2)
Load **only the chosen mode's reference files** — the list the router prints in
`load_references`. Heavy write protocols (processing, classification, provenance,
coverage) stay in `references/` and **never load during a QUERY**. The router
enforces this mechanically: a QUERY decision lists only `query_protocol.md`, and
the script exits non-zero if any write protocol leaks into a QUERY's list.

## 1. Canonical namespace (write only here)

**Input — wherever the map points (D5):** drawing PDFs may sit anywhere in the project
folder (`Specs & Drawings/Drawings/`, a Change Events folder, a submittal package).
Find them via `Claude/Map/`; never assume a fixed input folder; never move or rename
the raw PDF; record the source path as provenance.

**Output — D1 placement:** `Claude/Drawings/_spNN-name/drawing-db/<set>/` for
subproject sets (the normal case); flat `Claude/Drawings/drawing-db/` ONLY for a
genuinely job-wide set (none exist yet — that location is retired-empty). Writes
ONLY inside `Claude/`.

```
Claude/Drawings/_spNN-name/
└── drawing-db/                    ← canonical namespace (this skill only)
    └── <set-name>/
        ├── machine/               ← SOURCE OF TRUTH (schema-validated JSON)
        │   ├── manifest.json
        │   ├── sheet_index.json
        │   ├── sheet_classification.json
        │   ├── tag_index.json
        │   ├── cross_references.json
        │   ├── detail_index.json
        │   ├── coverage_status.json
        │   ├── markups.json          ← only if Bluebeam ingested
        │   └── source_validation.json
        ├── views/                 ← regenerable markdown projections (never hand-edited)
        │   ├── drawings.md            ← query entry point
        │   ├── rfi_candidates.md      ← D3 first-output contract
        │   ├── coordination_issues.md
        │   ├── open_questions.md
        │   ├── symbol_library.md
        │   ├── detail_index.md
        │   └── cards/[sheet].md
        └── sheets/                ← per-sheet pdf/png/txt + split manifest
```

Create the namespace before any processing:

```
python scripts/init_namespace.py --set-dir "<…/drawing-db/<set-name>>" --date YYYY-MM-DD
```

This is idempotent and safe: it never overwrites an existing `manifest.json` or
`coverage_status.json`. `<set-name>` should match how the set is named in
Procore/Bluebeam (e.g. `<set>-IFC`, `<set>-addendum-2`).

### Single source of truth (D5)
`machine/*.json` is the source of truth, schema-validated. `views/*.md` are
projections regenerated from the JSON — **never hand-edited.** Any consumer needing
a fact reads the JSON; markdown is for human/Claude reading only.

## 2. Provenance & coverage contracts (apply once the write engine lands)

Every extracted claim will carry (FS-3): `source_type`
(`pdf_text | bluebeam_text | bluebeam_markup | image_visual | crop | ocr | user_note | inference`),
`confidence` (`high | medium | low`), and `evidence` (sheet + page + the
callout/phrase or visual description). Inferences are labeled `[inferred]` and
never presented as fact. A sheet's `extraction_confidence` is a **confidence
ceiling** for every record sourced from it.

`coverage_status.json` is machine-readable (FS-4): total/processed counts,
pending/failed by discipline, and the question domains not yet answerable. **Every
consumer checks coverage before generating output**; partial coverage forces a
caveat or refusal, and every markdown view carries a coverage banner at top.

## Orientation & scope

The ClaudeOS MEMORY.md hierarchy is RETIRED (AGENTS.md §9) — do not look for
MEMORY.md files. Instead, before running:

1. Read `Claude/CLAUDE.md` → `_Memory/AGENTS.md` if not loaded this session.
   Scope context (design-firm split, subproject status, locked decisions) comes from
   `_Memory/` wiki pages via gse-wiki query.
2. Locate the source PDF's record in `Claude/Map/machine/manifest.json`; confirm
   the target subproject and `<set-name>` when ambiguous.
3. **Current-set check:** consult `Claude/Map/views/SUPERSEDED.md` + the subproject
   context before building on or answering from any set — never assume newest-modified
   is current (AGENTS.md §9).
4. **Design-firm rule:** classification and RFI-candidate outputs cite whose drawing
   a claim comes from (the responsible design firm). This job's firm roster and each
   firm's scope live in AGENTS.md §9; overlaps get flagged, never resolved.

## Backlinks + map-update contract (MANDATORY at end of BUILD/UPDATE)

1. **Backlinks (D9):** `views/drawings.md` and set-level views carry
   `**Wiki:** [[hub-page]]`; new sets get/update a wiki hub via gse-wiki ingest.
2. **Map-update contract** (`Claude/Map/machine/MAP_UPDATE_CONTRACT.md`): mark the
   source PDF's manifest record `processed` + `processed_home` → the new set dir.
   **Supersession discipline feeds the map:** when a new rev supersedes a set, keep
   this skill's internal supersession handling AND set the old raw record's status
   to `superseded` in the map manifest so `SUPERSEDED.md` stays the OS-wide safety
   rail. Regenerate map views (`gse-cartographer/scripts/render_views.py`).
3. Log the run in `_Memory/log.md` (house format).

## File safety
Before overwriting or renaming any existing file, show what changes and confirm
(GSE file-safety rule). When in doubt, write a new file rather than overwrite.
